What this is

v7 is the chart system Perplexity uses for social posts and research-blog figures. It's two surfaces, one rulebook. Social cards are editorial: one hero number leading, the rest as support. Research charts are instrument readouts: axes, gridlines, every value defensible. Same v7 tokens drive both — you flip treatment, not foundation.

Built to be picked up by a human or an agent. Tokens live in tokens.json. Every archetype is a live HTML embed below — resize your window and watch the cards respond.

Surfaces

#surfaces

Two axes: style (academic vs editorial) and value (dark vs light). A surface is a value flip. Hero and text invert with canvas; baseline and rule colors translate while preserving each surface's temperature. Slate for academic, warm grey for editorial.

#archetypes

Six archetypes cover the social-post grid. Each renders identically across all four surfaces. Only tokens change.

A
1:1 · 1080 × 1080
Stat + bar chart

Single hero stat with comparison bars. Includes Pareto bracket for highlighting a range. Use for: headline benchmark, accuracy peaks across budgets.

B
1:1 · 1080 × 1080
Pin spike

50-pin distribution with one hero peak. Inline callout chip names the peak. Use for: weekly highs, distribution snapshots, "broke every prior week."

C
4:5 · 1080 × 1350

Four metrics in a 2×2 grid. Each cell: stat + delta + sparkline. Use for: multi-benchmark quarterly comparison.

D
16:9 · 1920 × 1080

Wide dot-matrix distribution with one peak column. Use for: surface visualization, accuracy-vs-tokens across multiple suites.

E

F

04

Type rules

#type-rules

Non-negotiable. Every type decision must trace to a rule below.

Hero color
Hero stat sits in var(--text), never in accent. Color goes elsewhere. The hero earns its weight by typography, not by hue.
One accent
Maximum one accent per card. Two accents means neither is the accent. If a card uses teal, no other element can use teal.
Hero figures
Display heroes use proportional figures.
font-feature-settings: "pnum" 1, "lnum" 1, "kern" 1;
font-variant-numeric: proportional-nums lining-nums;
Stacked figures
Stacked columns (wallet lists, ledger rows) use tabular figures so digits align vertically.
font-feature-settings: "tnum" 1, "lnum" 1, "kern" 1;
font-variant-numeric: tabular-nums lining-nums;
Unit chrome
Unit glyphs (%, pp, pt, x) must never punch below the hero baseline. pp has descenders that, at full size, will collide with whatever sits below the hero (chart, chip, rule). Size down AND lift the baseline so the descender lands AT the hero baseline, not below it.
.t-hero .unit {
  font-size: 0.46em;
  line-height: 1;
  vertical-align: 0.34em;
  display: inline-block;
  font-variation-settings: 'wdth' 75, 'wght' 650, 'opsz' 60;
}
No em dashes
Use periods, commas, or separate sentences. Em dashes signal AI-generated copy.
No serif utility
Sans + Mono only for utility/tool UIs and labels. Serif (GT Canon) is reserved for editorial heroes.
Confidence in type
Confidence is encoded in typography. Weight, opacity, tracking carry the signal. No confidence badges, no certainty indicators. The type already says it.

Frame rules

#frame
PPLX mark
36×36 absolute top-right, ALWAYS. Every card. Color follows var(--text) so it inverts cleanly across surfaces.
Card padding
Every card uses --s-7 (56px) padding on all sides. No exceptions.
Even margins
Left and right margins must be equal. Non-negotiable. Catches 2px discrepancies.
Gradient fields
If any gradient effect is used, it spans the viewport as one unified field. Elements become semi-transparent windows into that field. No per-element gradient backgrounds.
Static over interactive
These cards are for posting. No CTAs, no hover states, no interaction logic. Carousels and draggable elements must earn their place. On static images, they don't.
07

Layout balancer

#balancer

Cards have fixed canvas, content has variable height. Static CSS can’t pick the right spacing strategy because it depends on how much content there actually is. fit.js measures content-to-canvas ratio (CCR) at runtime, classifies it, and applies one of five strategies via data-fit attribute and CSS variables. CSS owns the look; JS owns the strategy.

amplify
CCR < 0.40
Content too sparse. Grow the hero, expand hierarchy, widen section gaps.
distribute
CCR 0.40–0.70
Room available. Distribute remaining space proportionally with subtle hero growth.
anchor
CCR 0.70–0.95
Content fits comfortably. No adjustment. The locked design rhythm reads correctly.
pack
CCR 0.95–1.05
Content fills canvas. No adjustment. Content is exactly where it should be.
compact
CCR > 1.05
Content overflows. Tighten type, reduce row padding, shrink hero, dim auxiliary chrome.

Proof · same archetype, four content payloads

Same F-card archetype, same 1080×1920 canvas, four payloads (2, 5, 12, 18 wallet rows). The balancer picks a different strategy each time. Every result reads as a deliberately designed card.

Contract

Mark a card with data-fit-card and its content blocks with data-fit-content. fit.js writes data-fit and four CSS variables. CSS multiplies its own values by those variables.

// HTML
<section class="f-card surface-academic" data-fit-card>
  <header class="f-head" data-fit-content>...</header>
  <div class="f-goal" data-fit-content>...</div>
  <div class="f-list" data-fit-content>...</div>
</section>
<script src="./fit.js"></script>

/* CSS */
.f-row.hero .value {
  font-size: calc(120px * var(--fit-hero-scale, 1));
}
.f-row:not(.hero) {
  padding: calc(var(--s-3) * var(--fit-row-pad, 1)) 0;
}

The balancer never breaks the design contract. Mark stays 36×36, card padding stays 56px, spacing scale stays locked, type ramp stays locked. The balancer flexes only hero scale, row padding, and section gaps. Three knobs, five regimes, no other change to the system.

08

Edge cases

#edge-cases

The CCR balancer is a one-dimensional diagnostic. Three classes of content break it silently. Stress test, failure documentation, and the system response below.

Three failure classes the balancer cannot detect alone

horizontal overflow
A, D charts · F rows
Text wider than its column. CCR doesn't span horizontal axis. Long category labels collide silently while the balancer reports anchor.
density mismatch
B chart · D matrix
Sparse data inside an SVG looks empty, but DOM scrollHeight reports identical content. Balancer can't see visual emptiness.
missing structure
degenerate archetype
Knobs assume hero + list exist. On header-only cards, the knobs are no-ops. Resolved by archetype G.

Proof sheet · nine variants across three failure classes

Each row shows four content variants. The balancer's CCR and strategy badges populate live; verdict labels are color-coded by what actually went wrong.

The system response

No auto-mitigation. fit.js detects each failure class and emits a console warning plus a data-fit-warn attribute on the offending element. With <body class="fit-debug">, warnings render as a visible magenta strip on the card. Authors fix manually because the right response is a design decision, not a layout decision.

// Console output when content breaks an archetype contract
[pplx-fit] horizontal-overflow: category label "Compressed reasoning at 1.5K-tok budget" (37 chars) exceeds A-card limit of 12
[pplx-fit] density-mismatch: visual density 0.18 below B-card minimum 0.25 — chart will read as empty
[pplx-fit] missing-structure:list: archetype F requires block 'list' but no [data-fit-content="list"] found

// Each warning maps to an author fix:
//   horizontal-overflow → truncate, abbreviate, rotate, or pick a different archetype
//   density-mismatch    → add more data, raise the baseline, or swap archetype
//   missing-structure   → add the required block, or pick a different archetype (e.g. G)

Archetype G · editorial header

The third failure class ("missing structure") was resolved by adding a new archetype rather than retrofitting the existing six. G handles cards where the headline IS the content. Three sub-variants by character.

varianttitle facesize rangemax charsuse
G-display GT Canon Narrow Medium 72–180px 80 Single dramatic statement. Newspaper-style headline.
G-editorial GT Canon Regular 36–72px 240 Essay opener or longer headline. Reads as editorial type.
G-quote GT Canon Italic 48–96px 200 Attributed pull quote with Sans cite line.

G uses GT Canon as title face. PPLX Mono is reserved for kicker and footer only. The title is the hero on G — it gets the same display family that the hero stat uses on data cards. The original stress test wrongly used Mono for the header card title; the difference is visible in the proof sheet's fourth column.

Decision tree

#decision

Pick the archetype by the shape of your data, not by aesthetic preference. Start at the top.

Is your data a single hero number with supporting context?
├─ YES — one number plus 3 supporting metrics  → Archetype E (4:5 editorial stat block)
├─ YES — one number plus a comparison ranking → Archetype A (1:1 stat + bar)
├─ YES — before/after, A vs B, two states     → Archetype C (1:1 comparison pair)
└─ NO
   │
   Is your data a ranked list of 4-7 entries?
   ├─ YES — promote one to lead, story format  → Archetype F (9:16 wallet list)
   └─ NO
      │
      Is your data 4 independent metrics?
      ├─ YES — multi-metric quadrant            → Archetype B (1:1 2×2 metric grid)
      └─ NO
         │
         Is your data a curve or multi-category rhythm?
         ├─ YES — social card, shape over values  → Archetype D (16:9 dot-matrix)
         └─ YES — research figure, all values    → Research variant (multi-line plot)
            (when you need the actual axis-anchored numbers)

No data — just a claim, quote, or headline?
├─ Dramatic single statement                  → Archetype G-display
├─ Editorial framing with deck                → Archetype G-editorial
└─ Attributed pull quote                      → Archetype G-quote

Tie-breaker: if two archetypes fit, pick the one whose aspect ratio matches your distribution channel. Stories → 9:16. Feed posts → 1:1 or 4:5. Twitter/X cards → 16:9. Blog figures → always the research variant.

Agent path

#agent-path

A coding agent given a dataset and this page builds a card in three steps. No design system knowledge required — the contract lives in tokens.json and the decision tree picks the archetype.

01 · Fetch tokens

Pull surfaces, type ramp, spacing scale, and archetype contracts from tokens.json. Cache locally.

# single fetch, all tokens
$ curl tokens.json > v7.json
02 · Pick archetype

Use the decision tree to map your data shape to an archetype. The contract tells you which blocks are required.

# ranked list of 5? → F
$ jq '.archetypes.F' v7.json
03 · Render

Emit HTML using the contract’s required blocks. cqi units handle scaling. fit.js validates the result.

# validate after render
$ node fit.js card.html

Worked example — from a hub blog post on Sonar Pro’s SimpleQA F-score (0.858 vs Sonar baseline 0.773):

// 1. Data shape: hero number + 3 supporting metrics → Archetype E
// 2. Contract: kicker + title + hero + deck + triad + foot
// 3. Render — minimum viable card

<div class="card r-45 arche-e">
  <div class="head">
    <div class="h-kicker">Sonar Pro · SimpleQA</div>
    <div class="h-title">F-score lift vs Sonar baseline</div>
    <svg class="h-mark"><use href="#pp-mark"/></svg>
  </div>
  <div class="body">
    <div class="hero-stack">
      <div class="hero">0.858</div>
      <div class="deck">Sonar Pro beats Sonar by 11 points on SimpleQA factuality.</div>
    </div>
    <div></div>
    <div class="content">
      <div class="e-triad">
        <div class="e-cell"><div class="e-cell-label">vs Sonar</div><div class="e-cell-value">+11<span class="u">pp</span></div></div>
        <div class="e-cell"><div class="e-cell-label">Citations</div><div class="e-cell-value">2.0<span class="u">×</span></div></div>
        <div class="e-cell"><div class="e-cell-label">Copy AI</div><div class="e-cell-value">+20<span class="u">%</span></div></div>
      </div>
    </div>
  </div>
  <div class="foot"><span>SimpleQA · factuality</span><span>2026</span></div>
</div>

No SVG paths, no chart library, no rendering engine. The agent emits structural HTML, the CSS does the rest. The whole card is < 1.2KB before fonts.
